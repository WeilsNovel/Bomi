## 分支策略（详见 DECISIONS.md D005）

**模型**：特性分支 + 整合方合并。三端只往自己的 `feat/{端}-stageN` 提交，合并到 main 只能整合方做。

**5 道防线**（防乱根本机制）：
1. 包路径物理隔离（miniapp/admin/server/ai 目录不交叉，合并零冲突）
2. shared 整合方独占（唯一高风险文件单点控制）
3. 整合方是合并唯一仲裁者
4. 顺序合并：shared → server → 前端
5. Conventional Commits 前缀追溯归属

**分支命名**：三端对话接到 prompt 后第一动作 `git branch -m trae/agent-* feat/{端}-stageN`，禁止用 trae/agent-* 随机名工作。

**三端铁律**：
1. 只在 `feat/{端}-stageN` 提交，禁止碰 main
2. 禁止改 `packages/shared/`（提案给整合方）
3. 禁止跨端目录
4. 阶段完成向整合方报告，合并由整合方做

## 跨端协同铁律

1. 接口字段变更：先改 `shared/types` → 同步 server DTO → 同步前端 api，三处同提交
2. 错误码新增：先改 `shared/constants/error-code.ts` → server 使用 → 前端处理
3. 业务状态变更：先改 `shared/constants/business.ts` → server DB/逻辑 → 前端展示
4. 前端禁直连 AI 供应商，统一经 server `/api/ai/*` 转发
5. API Key 仅 `packages/ai/config/env.ts` 从 `process.env` 读取，禁止硬编码、禁止入 shared
